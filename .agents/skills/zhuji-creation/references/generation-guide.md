# 珠璣模式共用生成指南

珠璣模式以七個互補模組逐層展開角色。每個 Task 只產生一個模組；先讀目標 reference，再以該模組的責任邊界取捨內容。

## 分工

1. `appearance`：立即可觀察的外貌、聲音、服裝、感官與身體行為。
2. `inner_nature`：性格核心、價值、欲望、恐懼、矛盾、情緒與決策機制。
3. `extension`：背景、社會位置、能力、生活、人際網絡、與 `{{user}}` 的關係動態、性相關設定、私人空間與差異表現。
4. `trait_refinement`：把抽象標籤拆成來源、觸發、心理功能與外在行為。
5. `trait_dialogue`：依 5 至 8 個核心特質建立長篇代表語料及其人格結果。
6. `scene_dialogue`：依關係階段、場景、情緒與對象建立長篇角色語料。
7. `self_introduction`：角色第一人稱向 `{{user}}` 自我揭露，建立主觀自我認知。

## 共用品質規則

- 以 Blueprint 的核心概念與創作界線為最高內容範圍；不得為填滿模板而捏造會改變角色前提的重要事實。
- 保持模組正交：外顯不代寫內心史，內質不堆砌外貌，外延不重述全部性格，場景語料不加入旁白。
- 使用具體觸發、可觀察行為、語句節奏與情境差異，避免「溫柔、善良、漂亮」等無證據的通用形容詞堆疊。
- 描述性欄位不能只填標籤、程度詞或結論，例如「利他性：高」「思維模式：直覺型」「控制欲：中等」。應依欄位語意選取相關面向，交代形成原因或內在機制、觸發條件、具體表現、情境差異、對關係或後續行為的影響；不要求每欄機械填滿全部面向，但必須讓內容足以區分這名角色與套用同一標籤的其他角色。
- 適合舉例的欄位至少提供一個角色專屬的具體反應、選擇、語句或情境；例子必須支持設定，不得另行發明會改變 Blueprint 的重大事件。
- 姓名、年齡、日期、尺寸、分類、枚舉、固定稱謂等原子事實應保持精確簡短。詳細化只用於需要解釋的描述性內容，不得為了篇幅重複同義句、堆砌形容詞或把簡單事實改寫成空泛長文。
- 詳細內容仍須自然分句、段落清楚。優先增加資訊密度與角色專屬因果，不追求單句長度或總字數。
- 同一角色跨模組的年齡、身分、關係、外貌、價值與語氣必須一致。若既有正式內容與 Blueprint 衝突，停止並回報，不自行選邊。
- Accepted facts 是可追溯事實；Blueprint 或使用者決定是創作約束；Creator 補全是創作，不得混淆 provenance。
- `module.data` 直接使用目標 reference 定義的中文巢狀鍵值。所有必要頂層區塊都要存在，內容保持結構化；不要再把欄位壓進 Markdown `content/sections`。
- 所有新創作正文與語料預設使用正體中文。不得因角色是日本人、故事位於日本、採用日系文化或日文姓名，就混入日文原字、假名、羅馬字、日語語尾或未翻譯短句；只有使用者或已批准 Blueprint 明確要求該段使用外語時例外。正式姓名、必要專有名詞與 exact evidence 可保留原文。
- 模組按 `character → appearance → inner_nature → extension → trait_refinement → trait_dialogue → scene_dialogue → self_introduction` 依序生成；後一模組需維持前面正式模組的一致性。
- `expanded_extension` 是舊版 read-only module kind，不得在新 Proposal 或模式轉換目標中生成。
- 情色內容依 Blueprint 與人格設定落實，不把未要求的偏好、經歷或關係轉折當成既成事實。
- 不為角色或 greetings 分配 Token 預算。

## 提交前深度巡檢

提交前逐一掃描本次 `module.data` 的描述性葉節點：若刪除欄位名稱後，內容仍不能回答「這名角色為何如此、何時會出現、實際如何表現、會造成什麼差異」中的至少兩項，通常代表內容仍過於概括，應在不越過 Blueprint 與 provenance 邊界下補足。原子事實不套用此巡檢。
